# 과제정의서

## 1. 문서 개요

- 문서명: Local Copilot Chat 기능 과제정의서
- 작성일: 2026-05-07
- 기준 범위: 현재 저장소의 FastAPI 서버, 인증 서비스, 대화 저장소, 정적 웹 UI
- 문서 목적: 현재 구현 상태를 AI 및 IT 개발자 관점에서 정리하고, 유지보수 및 향후 확장 시 설계 기준으로 사용한다.

## 2. 과제 배경

GitHub Copilot 기반 대화 기능을 로컬 웹 애플리케이션에서 안정적으로 제공하려면 다음 문제를 함께 해결해야 한다.

- GitHub device flow 기반 로그인
- Copilot token 발급 및 갱신
- 브라우저 세션과 자격정보 바인딩
- 허용 모델 제어
- 서버 정본 기준 대화 복원
- GitHub 내부 usage 응답 형식 차이에 대한 best-effort 해석

기존의 단순 프록시 방식은 브라우저 자격정보 재사용 위험, 로그아웃 후 세션 정합성 문제, 모델 제어 부족, 대화 저장 범위 불명확성 같은 운영 리스크가 있었다. 본 과제는 FastAPI 서버 안에서 LiteLLM을 직접 호출하면서도, 사용자별 대화 보존과 안전한 인증 흐름을 함께 제공하는 로컬 채팅 시스템을 구현하는 것을 목표로 한다.

## 3. 과제 목표

- 별도 LiteLLM 프록시 없이 FastAPI 애플리케이션에서 직접 LiteLLM을 호출한다.
- 사용자가 브라우저에서 GitHub Copilot device flow 로그인을 완료할 수 있어야 한다.
- 브라우저에는 암호화된 credential envelope만 저장되어야 한다.
- envelope는 현재 브라우저의 세션 바인딩 검증을 통과해야만 사용 가능해야 한다.
- 로그인된 대화는 GitHub 사용자 ID 기준으로 서버 로컬 DB에 저장되어야 한다.
- 로그아웃 후 다시 로그인하거나, 같은 서버와 같은 DB를 사용하는 다른 기기에서 로그인해도 대화가 복원되어야 한다.
- 1주일이 지난 대화는 서버에서 자동으로 정리되어야 한다.
- 화면용 모델명과 실제 LiteLLM/Copilot 호출용 모델명을 분리해야 한다.
- Free와 Pro usage 응답 shape를 구분 처리해야 한다.

## 4. 대상 사용자 및 사용 맥락

- 대상 사용자: GitHub Copilot 계정으로 로컬 또는 내부망 환경에서 대화형 AI 기능을 사용하려는 개발자/운영자
- 사용 환경: 브라우저 기반 웹 UI
- 운영 형태: 단일 프로세스 로컬 실행 또는 소규모 내부 서버 실행
- 공유 범위: 동일 서버와 동일 SQLite DB를 공유하는 브라우저/기기 간 대화 복원 가능

## 6. 핵심 설계

### 6.1 인증과 자격정보

- GitHub device flow로 사용자 인증을 진행한다.
- access token으로 Copilot API token을 발급받는다.
- GitHub `/user`를 호출해 `github_user_id`, `github_login`을 확보한다.
- credential envelope에는 다음 핵심 필드를 포함한다.
  - `github_access_token`
  - `copilot_api_token`
  - `copilot_api_expires_at`
  - `copilot_api_base`
  - `credential_id`
  - `github_user_id`
  - `github_login`
- 브라우저에는 암호화된 envelope만 저장한다.
- envelope는 HttpOnly cookie에서 유도한 binding과 함께 검증한다.

### 6.2 대화 저장 구조

- 대화는 별도 SQLite DB에 저장한다.
- 저장 key는 일반화된 `scope_id`를 사용한다.
- 익명 상태:
  - 브라우저 cookie 기반 scope 사용
- 로그인 상태:
  - `user:<github_user_id>` 형식 사용
- 로그인 전 대화와 로그인 후 대화는 자동 병합하지 않는다.

### 6.3 로그아웃과 복원

- 로그아웃 시 현재 브라우저의 binding cookie를 회전한다.
- 같은 브라우저는 기존 envelope를 더 이상 사용할 수 없다.
- 서버에 저장된 사용자 대화는 삭제하지 않는다.
- 같은 GitHub 사용자로 다시 로그인하면 사용자 scope 기준 대화가 복원된다.
- 같은 서버와 같은 DB를 보는 다른 기기에서도 동일 사용자 대화를 복원할 수 있다.

### 6.4 대화 보존 기간

- 대화 보존 기준은 `conversations.updated_at`
- `updated_at < now - 7 days` 인 대화는 정리 대상
- read/write 경로에서 opportunistic cleanup 수행
- 대화 삭제 시 메시지는 `ON DELETE CASCADE`로 함께 삭제
- 비어 있는 scope는 정리

### 6.5 모델 구성

- `litellm_config.yaml`에서 `model_name`과 `litellm_params.model`을 함께 읽는다.
- `model_name`
  - UI 표시용
  - 클라이언트 선택용
  - 서버 allow-list 기준
- `litellm_params.model`
  - 실제 LiteLLM 호출용 provider model ID

예시:

```yaml
model_list:
  - model_name: gemini-3.1-pro
    litellm_params:
      model: gemini-3.1-pro-preview
```

### 6.6 usage 해석 전략

usage는 GitHub 내부 endpoint에 대한 best-effort 해석 결과다. 현재 서버는 세 가지 shape를 처리한다.

1. Free shape
   - `access_type_sku == free_limited_copilot`
   - `limited_user_quotas`
   - `monthly_quotas`

2. Pro / paid shape
   - `quota_snapshots`
   - `premium_interactions`
   - `chat.unlimited`

3. Generic fallback shape
   - `usage.remaining_chat_messages`
   - `usage.premium_requests_remaining`
   - `quotas.chat.remaining`
   - 기타 기존 score-based parser가 읽을 수 있는 구조

Free에서는 `chat` quota만 신뢰 가능하게 표시하고, `premiumRequests`를 직접 계산할 수 없으면 숨긴다.  
Pro에서는 `premium_interactions.remaining / entitlement`로 premium을 계산하고, `chat.unlimited = true`면 Chat을 무제한으로 해석한다.

## 7. 기능 요구사항

### FR-01. 정적 웹 채팅 화면 제공

- 시스템은 `/` 요청 시 단일 페이지 형태의 채팅 화면을 제공해야 한다.
- 화면에는 대화 목록, 새 대화 버튼, 모델 선택기, Copilot 상태 버튼, usage 영역, 메시지 입력창, 전송 버튼이 포함되어야 한다.
- 화면은 데스크톱과 좁은 뷰포트 모두에서 사용할 수 있어야 한다.

완료 기준:

- 루트 경로 접속 시 채팅 UI가 즉시 열린다.
- 대화 생성, 전환, 입력창 자동 높이 조정이 동작한다.

### FR-02. 허용 모델 목록 조회 및 강제

- 시스템은 `litellm_config.yaml`의 `model_name`을 기준으로 허용 모델 목록을 제공해야 한다.
- 서버는 `model_name` 기준으로 allow-list를 강제해야 한다.
- 실제 LiteLLM 호출 시에는 `litellm_params.model`을 사용해야 한다.

완료 기준:

- 모델 선택기에는 표시용 모델명이 노출된다.
- 실제 API 호출에는 provider model ID가 사용된다.
- 허용되지 않은 모델은 400으로 차단된다.

### FR-03. Copilot 로그인 상태 확인

- 시스템은 저장된 envelope의 유효성과 세션 바인딩 일치 여부를 확인해야 한다.
- 인증 성공과 usage 성공은 별개로 다뤄야 한다.
- status 응답은 필요 시 갱신된 envelope를 응답 헤더로 반환할 수 있어야 한다.

완료 기준:

- `POST /api/copilot/status`가 인증 여부, envelope 삭제 필요 여부, usage snapshot을 반환한다.
- 유효하지 않은 envelope는 200 응답과 `shouldClearEnvelope: true` 계약을 유지한다.

### FR-04. GitHub device flow 로그인 시작 및 완료

- 시스템은 GitHub device flow를 시작하고 사용자 코드와 인증 URL을 반환해야 한다.
- 로그인 완료 시 credential envelope와 usage snapshot을 반환할 수 있어야 한다.

완료 기준:

- `POST /api/copilot/login/start`가 `loginId`, `userCode`, `verificationUri`, `verificationUriComplete`, `expiresAt`, `nextPollAt`, `retryAfter`를 반환한다.
- 로그인 완료 시 `credentialEnvelope`, `credentialId`, `copilotTokenExpiresAt`, `usage`를 반환한다.

### FR-05. 로그인 폴링 및 백오프 제어

- 너무 이른 poll 요청은 상류 GitHub 호출 없이 자체 throttling 해야 한다.
- 오래된 `loginId`가 와도 최신 poll backoff가 유지되어야 한다.
- pending login 저장소는 프로세스 간 공유 가능한 SQLite여야 한다.

완료 기준:

- `POST /api/copilot/login/poll`이 `pending` 또는 `complete`를 반환한다.
- 여러 서비스 인스턴스가 같은 SQLite 파일을 공유해도 흐름이 유지된다.

### FR-06. credential envelope 생성 및 갱신

- 로그인 완료 시 access token과 Copilot token을 포함하는 암호화 envelope를 생성해야 한다.
- Copilot token 만료 임박 시 서버는 자동으로 갱신해야 한다.
- 갱신된 envelope는 응답 헤더 `X-Copilot-Credential-Envelope`로 전달되어야 한다.

완료 기준:

- 브라우저에는 암호화된 envelope만 저장된다.
- 만료 임박 envelope 사용 시 서버가 새 envelope를 돌려준다.

### FR-07. 브라우저 세션 바인딩 검증 및 로그아웃

- envelope는 현재 브라우저의 binding cookie와 함께 검증해야 한다.
- 로그아웃 시 binding cookie를 회전해야 한다.
- 현재 브라우저는 재인증 전 기존 envelope를 재사용할 수 없어야 한다.

완료 기준:

- 로그아웃 후 같은 브라우저가 기존 envelope를 계속 보내면 인증 실패 처리된다.
- 서버는 별도 revocation list 없이 cookie rotation만으로 현재 브라우저를 차단한다.

### FR-08. 사용자별 대화 저장 및 복원

- 로그인된 사용자는 GitHub 사용자 ID 기준으로 대화가 저장되어야 한다.
- 같은 사용자로 다시 로그인하면 대화가 복원되어야 한다.
- 같은 서버/DB를 보는 다른 브라우저에서도 동일 사용자 대화가 복원되어야 한다.
- 익명 대화는 브라우저 세션 scope 기준으로 분리 저장되어야 한다.
- 익명 대화와 로그인 후 대화는 자동 병합하지 않아야 한다.

완료 기준:

- 재로그인 후 동일 사용자 대화가 복원된다.
- 다른 브라우저/기기에서도 동일 사용자 대화가 복원된다.
- 익명 대화는 로그인 사용자 대화와 분리된다.

### FR-09. 대화 TTL 정리

- 7일이 지난 대화는 자동 정리되어야 한다.
- 정리 기준은 마지막 대화 업데이트 시각이다.

완료 기준:

- 7일을 초과한 대화는 read/write 시점에 제거된다.
- 최신 대화는 유지된다.

### FR-10. 채팅 요청 검증 및 검색 opt-in

- 채팅 요청 시 메시지, 모델, `searchMode`를 검증해야 한다.
- `searchMode`는 `auto`, `off`, 생략만 허용한다.
- 명시적 검색 요청일 때만 DuckDuckGo 검색을 수행해야 한다.
- 검색 결과는 lower-trust reference 메시지로만 주입해야 한다.
- 검색 실패 시 ordinary chat으로 fallback 해야 한다.

완료 기준:

- 인증 없는 채팅 요청은 401
- 잘못된 모델 / 메시지 / searchMode는 400
- 검색 실패 시 채팅 흐름은 유지

### FR-11. SSE 기반 스트리밍 응답

- 채팅 응답은 SSE로 스트리밍해야 한다.
- 사용자는 전송 버튼의 `중지` 상태로 스트림을 중단할 수 있어야 한다.
- 실패 시 안정적인 SSE 에러 payload와 `[DONE]`를 유지해야 한다.

완료 기준:

- 스트리밍 성공 시 delta가 순차적으로 표시된다.
- abort 시 즉시 중단 상태가 반영된다.
- model-not-supported, rate limit 등은 전용 메시지로 전달된다.

### FR-12. usage snapshot 표시

- usage는 GitHub 내부 endpoint를 best-effort로 조회해야 한다.
- Free 계정은 chat quota만 확실히 표시하고, premium을 계산할 수 없으면 premium 영역을 숨길 수 있어야 한다.
- Pro 계정은 premium remaining / total / used를 계산할 수 있어야 한다.
- `chat.unlimited = true`일 경우 Chat을 `무제한`으로 표시해야 한다.

완료 기준:

- Free 계정은 chat 수치가 표시된다.
- Free 계정에서 premium 근거가 없으면 premium 영역이 숨겨진다.
- Pro 계정은 premium `remaining / total / used`를 표시할 수 있다.
- usage 실패가 인증 실패로 오인되지 않는다.

## 8. 비기능 요구사항

### 보안

- 브라우저에는 평문 GitHub token / Copilot token을 저장하지 않는다.
- envelope는 서버 시크릿 기반 대칭 암호화로 보호한다.
- binding cookie는 `HttpOnly`, `SameSite=Strict`를 가져야 한다.
- HTTPS 환경에서는 `Secure` 쿠키 사용을 지원해야 한다.

### 신뢰성

- pending/complete login 상태는 SQLite로 일관되게 관리되어야 한다.
- 완료 직후 응답 유실에 대비해 replay가 가능해야 한다.
- usage 응답 shape 변화에 대해 graceful degradation 해야 한다.

### 성능 및 UX

- 첫 화면은 정적 파일로 즉시 열려야 한다.
- 응답은 스트리밍 방식으로 표시되어야 한다.
- 로그인 성공과 usage 성공은 별개로 안내되어야 한다.

### 운영성

- 주요 동작은 환경변수로 조정 가능해야 한다.
- 기본 실행은 단일 프로세스 로컬 서버로 가능해야 한다.
- 로컬 검증용 테스트 스크립트 또는 집중 테스트로 핵심 시나리오를 확인할 수 있어야 한다.

## 9. 데이터 저장 및 처리 정책

### 브라우저 측

- `localStorage`: 암호화된 `credentialEnvelope`
- 메모리 상태: 대화 목록, 활성 대화, 스트리밍 상태, 로그인 진행 상태, usage snapshot

### 서버 측

- HttpOnly cookie: 현재 브라우저 세션 바인딩 값
- SQLite (auth): pending login, completed login, poll backoff, replay TTL
- SQLite (chat): `scope_id` 기준 대화 목록, 활성 대화, visible transcript, 모델

### 처리 원칙

- GitHub `/user.id`는 사용자 대화 scope의 안정 키로 사용한다.
- `credential_id`는 display/debug 식별자일 뿐 사용자 저장 키가 아니다.
- visible transcript 외 인증 정보, token, system/reference 메시지는 저장하지 않는다.
- 7일이 지난 대화는 자동 정리한다.

## 11. 검증 기준

### 집중 테스트 기준

- GitHub device flow form-urlencoded 요청 검증
- poll throttling/backoff 검증
- completed login replay 검증
- envelope binding mismatch 검증
- 사용자별 대화 복원 검증
- 로그아웃 후 재로그인 복원 검증
- 다른 클라이언트에서 동일 사용자 복원 검증
- 익명 대화 비병합 검증
- 7일 TTL 정리 검증
- 모델 표시용 ID와 provider model ID 분리 검증
- model-not-supported SSE 에러 메시지 검증
- Free usage shape / Pro quota_snapshots shape 정규화 검증

### 수동 확인 기준

- 브라우저 첫 접속 시 Copilot 상태 확인 가능
- 로그인 시작 후 GitHub 인증 창이 열리고 user code 표시
- 인증 완료 후 연결 상태 전환
- Free 계정에서 chat usage 표시, premium 영역 숨김 가능
- Pro 계정에서 chat 무제한 및 premium requests 표시 가능
- 메시지 전송 시 응답이 점진적으로 표시
- 로그아웃 후 현재 브라우저는 인증 해제되지만, 재로그인 시 사용자 대화 복원

## 12. 성공 기준

다음 조건을 만족하면 현재 단계 목표를 달성한 것으로 본다.

- 로컬에서 서버를 실행하고 브라우저로 접속해 기본 UI를 사용할 수 있다.
- GitHub Copilot 로그인부터 채팅 응답 수신까지 정상 동작한다.
- 같은 GitHub 사용자로 다시 로그인하면 대화가 복원된다.
- 같은 서버/DB를 보는 다른 브라우저에서도 동일 사용자 대화가 복원된다.
- 7일이 지난 대화는 정리된다.
- 화면용 모델명과 실제 provider model명이 분리되어 동작한다.
- Free와 Pro usage 응답이 각각 의미 있는 방식으로 해석된다.
- usage 실패 또는 partial 상태가 로그인 실패로 오인되지 않는다.

## 13. 결론

본 프로그램은 로컬 웹 UI에서 GitHub Copilot 기반 채팅을 안전하게 제공하기 위한 경량 애플리케이션이다. 핵심 가치는 GitHub device flow 로그인, 암호화된 credential envelope, 브라우저 세션 바인딩, GitHub 사용자 ID 기반 대화 영속화, SQLite 기반 저장소, LiteLLM 직접 호출, SSE 스트리밍 응답, 그리고 Free/Pro 응답 shape를 구분한 best-effort usage 표시를 하나의 흐름으로 통합한 점에 있다. usage 정보는 여전히 GitHub 내부 endpoint에 의존하므로 보조 지표로 취급하며, 우선순위는 로그인과 채팅 경로의 안정성, 안전성, 그리고 대화 복원 경험 유지에 둔다.
