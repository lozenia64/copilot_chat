# 과제정의서

## 1. 문서 개요

- 문서명: Local Copilot Chat 기능 과제정의서
- 작성일: 2026-05-04
- 기준 범위: 현재 저장소에 구현된 FastAPI 서버, 인증 서비스, 정적 웹 UI, 테스트 코드
- 문서 목적: 본 문서는 현재 프로그램이 해결하려는 문제, 제공해야 하는 기능, 운영 조건, 보안 요구사항, 검증 기준을 명확히 정의하기 위해 작성한다.

## 2. 과제 배경

GitHub Copilot 기반 대화 기능을 로컬 웹 애플리케이션에서 사용하려면 인증, 토큰 관리, 모델 제한, 스트리밍 응답 처리, 브라우저 세션 보안이 함께 해결되어야 한다. 기존의 단순 프록시 방식은 브라우저 자격정보 재사용 위험, 로그인 상태 관리 복잡성, 모델 통제 부족 같은 문제가 있다.

본 과제는 LiteLLM Python 라이브러리를 직접 사용하면서도, 각 브라우저 세션에 종속된 안전한 Copilot 로그인과 로컬 채팅 UI를 제공하는 프로그램을 구현하는 데 목적이 있다.

## 3. 과제 목표

- 별도 LiteLLM 프록시 서버 없이 FastAPI 애플리케이션에서 직접 LiteLLM을 호출한다.
- 사용자가 브라우저에서 GitHub Copilot device flow 로그인 절차를 완료할 수 있어야 한다.
- 로그인 완료 후 발급된 Copilot 관련 자격정보는 암호화된 형태로만 브라우저에 저장되어야 한다.
- 자격정보는 현재 브라우저 세션에 바인딩되어 다른 브라우저나 다른 세션에서 재사용될 수 없어야 한다.
- 로그인된 계정의 usage snapshot은 서버가 best-effort로 조회해 UI에 표시하되, 조회 실패나 부분 성공이 인증 실패로 간주되지는 않아야 한다.
- 허용된 모델 목록만 UI에서 선택 가능해야 하며 서버에서도 동일한 allow-list를 강제해야 한다.
- 채팅 응답은 실시간 스트리밍으로 사용자에게 전달되어야 한다.
- 토큰 만료 임박 시 서버가 자동으로 갱신하고, 갱신된 자격정보를 브라우저에 재전달해야 한다.

## 4. 대상 사용자 및 사용 맥락

- 대상 사용자: 로컬 PC에서 GitHub Copilot 계정으로 대화형 AI 기능을 사용하려는 개발자 또는 운영자
- 사용 환경: 단일 사용자가 로컬 브라우저에서 접속하는 웹 UI
- 운영 형태: 개인 또는 소규모 내부 도구 수준의 로컬 실행 환경

## 6. 핵심 기능 정의

### FR-01. 정적 웹 채팅 화면 제공

- 시스템은 `/` 요청 시 단일 페이지 형태의 채팅 화면을 제공해야 한다.
- 화면에는 대화 목록, 새 대화 생성 버튼, 모델 선택기, Copilot 상태 버튼, 메시지 입력창, 전송 버튼이 포함되어야 한다.
- 화면은 데스크톱과 좁은 뷰포트 모두에서 사용할 수 있도록 반응형으로 동작해야 한다.

완료 기준:

- 사용자가 브라우저에서 루트 경로로 접속하면 채팅 UI가 표시된다.
- 새 대화 생성, 대화 전환, 입력창 자동 높이 조정이 동작한다.

### FR-02. 허용 모델 목록 조회 및 강제

- 시스템은 `litellm_config.yaml`에 정의된 모델 이름을 기반으로 허용 모델 목록을 제공해야 한다.
- 클라이언트는 `GET /api/models` 결과를 모델 선택기에 반영해야 한다.
- 서버는 `POST /api/chat` 요청 시 허용 목록에 없는 모델을 거부해야 한다.

완료 기준:

- 모델 목록 API가 `data[].id` 형식으로 모델 ID를 반환한다.
- 허용되지 않은 모델 요청은 400 오류로 차단된다.

### FR-03. Copilot 로그인 상태 확인

- 시스템은 브라우저가 보유한 암호화 credential envelope의 유효성을 확인할 수 있어야 한다.
- 서버는 현재 브라우저 세션에 바인딩된 쿠키를 기준으로 envelope를 검증해야 한다.
- 인증 상태와 usage snapshot 조회 결과는 분리해서 다뤄야 한다.
- usage snapshot은 서버가 GitHub 내부 endpoint에 대해 best-effort로 조회한 결과여야 하며, 인증 성공 상태에서도 `ok`, `partial`, `unavailable` 중 하나가 될 수 있다.
- 유효하지 않거나 세션 바인딩이 맞지 않는 경우 클라이언트가 저장된 envelope를 삭제할 수 있도록 명시적인 응답을 내려야 한다.

완료 기준:

- `POST /api/copilot/status`가 인증 여부, envelope 삭제 필요 여부, 만료 임박 여부, usage snapshot을 반환한다.
- `authenticated: true` 이면서 `usage.status`가 `partial` 또는 `unavailable` 인 응답을 허용한다.
- 유효하지 않은 envelope는 브라우저에서 제거된다.

### FR-04. GitHub device flow 로그인 시작

- 시스템은 GitHub device flow를 시작하고 사용자 코드와 인증 URL을 반환해야 한다.
- 로그인 시작 시 현재 브라우저 세션에 대한 바인딩 쿠키가 없으면 새로 발급해야 한다.
- 로그인 진행 상태는 서버 측 저장소에 보관되어야 한다.

완료 기준:

- `POST /api/copilot/login/start`가 `loginId`, `userCode`, `verificationUri`, `verificationUriComplete`, `expiresAt`, `nextPollAt`, `retryAfter`를 반환한다.
- 클라이언트는 받은 URL을 새 창으로 열 수 있다.

### FR-05. 로그인 폴링 및 백오프 제어

- 시스템은 로그인 진행 상태를 폴링 방식으로 확인할 수 있어야 한다.
- 서버는 너무 이른 폴링 요청을 상류 GitHub API 호출 없이 자체적으로 throttling 해야 한다.
- 로그인 핸들은 회전 가능한 `loginId` 형태여야 하며, 이전 핸들이 재전송되더라도 최신 백오프 상태가 우선되어야 한다.
- 로그인 상태 저장소는 프로세스 간 공유를 고려해 SQLite 기반이어야 한다.

완료 기준:

- `POST /api/copilot/login/poll`이 `pending` 또는 `complete` 상태를 반환한다.
- 오래된 `loginId`로 재요청해도 최신 poll backoff가 적용된다.
- 여러 서비스 인스턴스가 같은 SQLite 파일을 공유해도 흐름이 유지된다.

### FR-06. Copilot 자격정보 생성 및 암호화 저장

- 로그인 완료 시 시스템은 GitHub access token으로 Copilot API token을 발급받아야 한다.
- 해당 자격정보는 암호화된 credential envelope로 변환되어야 한다.
- 브라우저에는 원문 토큰이 아닌 암호화된 envelope만 저장되어야 한다.
- 로그인 완료 응답은 가능하면 usage snapshot을 함께 반환하되, 조회 불가 시에도 로그인 완료 자체는 성공으로 처리할 수 있어야 한다.

완료 기준:

- 로그인 완료 응답에 `credentialEnvelope`, `credentialId`, `copilotTokenExpiresAt`가 포함된다.
- 로그인 완료 응답의 `usage` 는 `ok`, `partial`, `unavailable` 중 하나일 수 있다.
- 브라우저는 envelope를 `localStorage`에 저장한다.

### FR-07. 브라우저 세션 바인딩 및 재사용 방지

- 시스템은 credential envelope를 현재 브라우저 세션에만 유효하도록 바인딩해야 한다.
- 세션 바인딩은 HttpOnly 쿠키와 서버 시크릿으로부터 파생된 바인딩 값으로 검증해야 한다.
- 로그아웃 시 바인딩 쿠키를 회전시켜 이전 envelope를 무효화해야 한다.

완료 기준:

- 다른 세션 또는 회전된 세션 쿠키로는 기존 envelope를 사용할 수 없다.
- 로그아웃 후 기존 envelope는 인증 실패 처리된다.

### FR-08. 채팅 요청 검증 및 실행

- 시스템은 채팅 요청 시 메시지 배열과 모델 값을 검증해야 한다.
- 메시지는 최소 1개 이상이어야 하며, 각 메시지는 유효한 `role`과 `content`를 가져야 한다.
- 인증되지 않은 경우 채팅 요청은 거부되어야 한다.

완료 기준:

- `POST /api/chat`는 인증 없는 요청에 401을 반환한다.
- 잘못된 모델 또는 잘못된 메시지 형식 요청은 400을 반환한다.

### FR-09. SSE 기반 스트리밍 응답 제공

- 시스템은 채팅 응답을 SSE 형식으로 스트리밍해야 한다.
- 클라이언트는 수신한 delta를 누적하여 화면에 즉시 반영해야 한다.
- 스트리밍 중 사용자가 중지 버튼을 누르면 요청을 abort 할 수 있어야 한다.
- 스트리밍 시작 실패 또는 중간 오류 시 SSE 에러 payload를 보내고 종료 신호를 제공해야 한다.

완료 기준:

- 응답 헤더가 `text/event-stream`으로 제공된다.
- 스트리밍 성공 시 delta가 순차적으로 표시된다.
- 오류 시 클라이언트가 메시지를 표시하고 스트림이 종료된다.

### FR-10. Copilot 토큰 자동 갱신

- 서버는 Copilot API token 만료가 임박한 경우 채팅 요청 전에 자동으로 토큰을 갱신해야 한다.
- 갱신된 credential envelope는 응답 헤더를 통해 브라우저에 전달되어야 한다.

완료 기준:

- 만료 임박 envelope로 채팅 요청 시 서버가 새 토큰을 발급한다.
- 응답 헤더 `X-Copilot-Credential-Envelope`에 갱신된 값이 포함된다.

### FR-11. 클라이언트 대화 세션 관리

- 사용자는 여러 개의 로컬 대화를 생성하고 전환할 수 있어야 한다.
- 각 대화는 첫 번째 사용자 메시지를 기반으로 제목이 자동 생성되어야 한다.
- 대화 이력은 현재 페이지 메모리 내에서만 관리한다.

완료 기준:

- 새 대화 생성, 목록 렌더링, 활성 대화 전환이 가능하다.
- 페이지를 새로고침하면 대화 목록은 초기화된다.

### FR-12. UI 편의 기능 제공

- 사용자는 빠른 프롬프트 버튼, 코드 블록 복사 버튼, 로그인 상태 모달, 토스트 메시지를 사용할 수 있어야 한다.
- 메시지 렌더링은 기본적인 문단, 굵게, 기울임, 인라인 코드, 코드 블록 표현을 지원해야 한다.

완료 기준:

- 코드 블록 복사 기능이 동작한다.
- 로그인 상태와 오류가 모달 및 토스트로 표시된다.

## 7. 비기능 요구사항

### 보안

- 자격정보는 평문으로 브라우저 저장소에 저장하지 않는다.
- envelope 암호화에는 서버 시크릿 기반 대칭 암호화를 사용한다.
- 세션 바인딩 쿠키는 `HttpOnly`, `SameSite=Strict` 속성을 가져야 한다.
- HTTPS 환경에서는 설정을 통해 `Secure` 쿠키를 사용할 수 있어야 한다.
- 인증 및 세션 불일치 시 명확한 오류 코드와 메시지를 반환해야 한다.
- 일반 런타임 사용 시 사용자 GitHub token 또는 Copilot token을 `.env` 에 저장하도록 요구하지 않는다.

### 신뢰성

- GitHub device flow의 pending 상태와 완료 직후 상태를 서버 측에서 일관되게 추적해야 한다.
- 동일 호스트에서 여러 서비스 프로세스가 동작해도 SQLite 저장소를 통해 로그인 폴링 상태가 유지되어야 한다.
- 완료 직후 응답 유실에 대비해 제한 시간 동안 완료 결과 재전달이 가능해야 한다.
- usage snapshot 조회는 상류 응답 형식 변화나 일시 오류에 대해 graceful degradation 해야 하며, 확인 가능한 값만 노출해야 한다.

### 성능 및 사용자 경험

- 첫 화면은 별도 SPA 빌드 없이 정적 파일로 즉시 열려야 한다.
- 응답은 전체 완료 후 반환이 아니라 스트리밍 방식으로 표시되어야 한다.
- 사용자가 스트리밍을 중단하면 불필요한 대기 없이 중단 상태를 인지할 수 있어야 한다.
- 로그인 성공과 usage 표시 성공은 별개로 안내되어야 하며, usage 조회 실패만으로 로그인 실패 메시지를 보여주지 않아야 한다.

### 운영성

- 주요 동작은 환경변수로 조정 가능해야 한다.
- 기본 실행은 단일 프로세스 로컬 서버로 가능해야 한다.
- 테스트 코드로 핵심 인증 흐름과 스트리밍 계약을 검증할 수 있어야 한다.

## 8. 데이터 저장 및 처리 정책

### 브라우저 측

- `localStorage`: 암호화된 `credentialEnvelope` 저장
- 메모리 상태: 대화 목록, 활성 대화, 현재 스트리밍 상태, 로그인 진행 상태

### 서버 측

- HttpOnly 쿠키: 현재 브라우저 세션 바인딩 식별용 값 저장
- SQLite: pending login, completed login, poll backoff, replay TTL 관리
- 메모리: 앱 인스턴스 수준 서비스 객체 및 런타임 설정

### 데이터 처리 원칙

- 브라우저에는 GitHub access token 또는 Copilot API token 평문을 직접 저장하지 않는다.
- 완료된 로그인 정보는 필요한 최소 시간 동안만 replay 용도로 보관한다.
- 대화 내용은 현재 구현 기준 서버에 영구 저장하지 않는다.

## 9. 설정 요구사항

### 기본 런타임 설정

- `.env.example` 은 최소 설정 기준을 반영해야 한다.
- 대부분의 실행 환경에서는 `COPILOT_ENVELOPE_SECRET` 설정을 권장한다.
- `DEFAULT_MODEL` 은 기본 모델을 바꾸고 싶을 때만 선택적으로 사용한다.

### 추가 조건

- `.env` 파일이 존재할 경우 애플리케이션 시작 시 자동으로 로드되어야 한다.
- `COPILOT_ENVELOPE_SECRET`가 없으면 임시 시크릿으로 실행 가능하되, 서버 재시작 후 기존 envelope는 무효화된다.
- 일반 런타임 인증을 위해 사용자 GitHub token 또는 Copilot API token을 `.env` 에 넣지 않는다.

### 고급 override 범위

- 필요 시 SQLite 경로, 쿠키 속성, 완료 로그인 TTL, 토큰 refresh skew, GitHub device flow endpoint, Copilot API key endpoint, usage endpoint 목록을 환경변수로 조정할 수 있어야 한다.
- 위 항목은 디버깅 또는 특수 운영 환경용이며, 기본 로컬 실행의 선행조건으로 간주하지 않는다.

## 10. 검증 기준

### 자동화 테스트 기준

- GitHub device flow 시작 요청이 form-urlencoded로 전송되는지 검증한다.
- access token polling 요청이 form-urlencoded로 전송되는지 검증한다.
- 너무 이른 poll 요청이 상류 호출 없이 throttling 되는지 검증한다.
- 오래된 `loginId` 요청에도 최신 authoritative backoff가 유지되는지 검증한다.
- 여러 서비스 인스턴스가 같은 pending login 저장소를 공유할 수 있는지 검증한다.
- 완료된 로그인 결과가 현재 및 이전 핸들에 대해 재생성 없이 replay 되는지 검증한다.
- 로그아웃 후 기존 envelope가 무효화되는지 검증한다.
- status 및 login completion 응답에서 usage snapshot이 정규화되어 반환되는지 검증한다.
- usage endpoint fallback, partial metric 처리, unavailable 처리 동작을 검증한다.
- 토큰 갱신 시 새 envelope가 응답 헤더에 포함되는지 검증한다.
- LiteLLM 스트리밍 시작 실패 시 SSE 에러 계약이 유지되는지 검증한다.

### 수동 확인 기준

- 브라우저 첫 접속 시 Copilot 상태 확인이 가능하다.
- 로그인 시작 후 GitHub 인증 창이 열리고 user code가 표시된다.
- 인증 완료 후 상태가 연결됨으로 전환된다.
- usage 표시는 GitHub 응답 상태에 따라 정상, 일부만 확인, 확인 불가 중 하나로 보일 수 있다.
- 메시지 전송 시 응답이 점진적으로 표시된다.
- 스트리밍 중지 버튼이 동작한다.
- 로그아웃 시 재인증 전에는 채팅이 불가능하다.

## 11. 성공 기준

다음 조건을 만족하면 본 과제의 현재 단계 목표를 달성한 것으로 본다.

- 로컬에서 서버를 실행하고 브라우저로 접속해 기본 UI를 사용할 수 있다.
- GitHub Copilot 로그인부터 채팅 응답 수신까지 단일 브라우저 세션에서 끊김 없이 완료된다.
- 허용된 모델만 선택 및 호출 가능하다.
- 자격정보는 암호화되어 저장되며 세션 바인딩 검증을 통과해야만 사용된다.
- usage snapshot은 서버가 best-effort로 표시하며, `ok` 가 아니더라도 로그인과 채팅이 유지될 수 있다.
- 토큰 만료 임박 상황에서도 자동 갱신 후 채팅이 지속된다.
- 핵심 인증 흐름과 스트리밍 계약이 자동화 테스트로 검증된다.

## 12. 향후 확장 고려사항

현재 구현 범위를 넘어서지만, 추후 확장 후보는 다음과 같다.

- 대화 내역 영속 저장 및 복원
- 파일 첨부 또는 이미지 입력 UI
- 사용자별 설정 저장
- 다중 사용자 또는 팀 단위 세션 관리
- 모델별 옵션 설정 UI
- 운영 로그 및 관리자 진단 화면

## 13. 결론

본 프로그램은 로컬 웹 UI에서 GitHub Copilot 기반 채팅을 안전하게 제공하기 위한 경량 애플리케이션이다. 핵심 가치는 단순한 채팅 인터페이스 자체보다, GitHub device flow 로그인, 암호화된 credential envelope, 브라우저 세션 바인딩, SQLite 기반 poll coordination, LiteLLM 직접 호출, SSE 스트리밍 응답, 그리고 best-effort usage snapshot 표시를 하나의 흐름으로 통합한 점에 있다. 다만 usage 정보는 GitHub 내부 endpoint에 의존하므로 보조 지표로 취급하며, 본 과제의 성공은 로그인과 채팅 경로의 안정성 및 안전성에 우선적으로 둔다.
